## 2. 前端后续路线

项目为单人维护、自用部署，原则是优先降低维护成本，不为了框架而迁移框架。

### 阶段 A：逐页完善信息架构

1. **资源搜索**
   - 海报瀑布流与紧凑列表双模式；
   - 资源质量、文件数量、更新时间和有效性可视化；
   - 快速预览与创建订阅合并为单一操作流。
2. **订阅管理**
   - 海报视图与数据表格双模式；
   - 独立订阅详情页；
   - 剧集网格、缺集状态和自动化流水线。
3. **网盘与下载**
   - 面包屑、批量操作和任务状态进一步统一；
   - 转存、重命名、STRM、Aria2 的关联状态可视化。
4. **设置中心**
   - 改为“连接、自动化、命名规则、通知、维护”任务型分组；
   - 增加配置完成度和连接测试总览。

### 阶段 B：保留 Alpine.js，拆分前端模块

```text
static/js/
  app.js
  core/
    api.js
    router.js
    formatters.js
    notifications.js
  stores/
    subscriptions.js
    settings.js
    jobs.js
    downloads.js
    drive.js
  features/
    dashboard.js
    search.js
    subscriptions.js
    drive.js
    downloads.js
    settings.js
```

优先抽离：

1. `apiFetch()` 与统一错误处理；
2. 时间、容量、速度格式化；
3. 下载任务；
4. 网盘管理；
5. 在线更新。

暂不引入 React/Vue/Vite。只有在需要复杂客户端路由、组件库或多人协作时再重新评估。

## 3. 后端架构路线

### 短期

- 按业务能力拆分 `subscriptions.rs`、`drive.rs`、`worker.rs` 等大型文件。
- 为订阅批量检查增加批量更新/单次落盘能力。
- 给 JSON 数据增加显式 `schema_version` 和迁移函数。
- 统一 API 响应和前端请求处理。

### 中期

- 抽象 `CloudDriveProvider`，隔离分享探测、转存、目录浏览和下载地址获取。
- 用领域事件统一 Job、Notification、Push 和 Metrics。
- 将不同 `JobKind` 拆成独立 handler。

### 数据存储原则

当前继续使用 JSON。只有出现以下情况时再迁移 SQLite：

- 数百个以上订阅；
- 长期保存大量任务历史；
- JSON 文件达到数十 MB；
- 需要复杂筛选、统计或索引查询。

## 4. 功能优先级

### P1：订阅详情与缺集检测

- 剧集网格；
- 已发现、已转存、已下载、已生成 STRM 状态；
- 缺集检测与一键补集；
- 单订阅任务历史。

### P2：媒体更新日历

- 今日更新；
- 本周待更新；
- 延迟更新；
- 已到完结日期但仍缺集。

### P3：资源质量评分与自动换源

评分维度包括清晰度、完整性、季匹配、命名规范、更新时间、广告和合集风险。

### P4：自动化流水线视图

```text
发现更新 → 文件过滤 → 版本选择 → 夸克转存 → 重命名
         → STRM → Aria2 → 通知
```

每一步显示成功、跳过、失败和重试状态。

### P5：PWA

- 添加到手机桌面；
- 独立窗口；
- 静态资源缓存；
- 快速检查订阅和查看下载进度。
