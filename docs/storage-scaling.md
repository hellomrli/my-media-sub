# JSON Store 性能基线与 SQLite 决策

## 当前策略

业务 Store 继续使用带 `schema_version` 的紧凑 JSON 信封、原子临时文件、`fsync` 和 rename。运行时记录每个 Store 的当前大小、读写次数与字节数、解析/写入累计耗时和失败数，并在系统诊断页展示。

现有保留上限：

| 数据 | 保留策略 |
|---|---|
| 订阅检查历史 | 每订阅最近 30 条 |
| 换源历史 | 每订阅最近 50 条 |
| 历史分享链接 | 每订阅最近 50 条 |
| 通知 | 最近 300 条 |
| Job | 最近 500 条 |
| 自动化事件 | 普通 30 天、失败 90 天，最多 5,000 条 |

订阅和 Job 使用 ID → 数组位置内存索引；自动化事件按订阅、correlation ID 和 Job ID 建立索引。`POST /api/storage/compact` 会重新应用所有保留策略，并将全部 Store 改写为紧凑 JSON。

## 2026-07-11 性能基线

测试：`tests/json_performance.rs`。Debug 测试构建、当前开发机数据如下，仅用于数量级判断，不作为跨硬件 SLA。

| Store | 记录数 | 紧凑大小 | 序列化 | 解析 |
|---|---:|---:|---:|---:|
| subscriptions | 500 | 103,120 B | 2.63 ms | 2.30 ms |
| jobs | 10,000 | 2,678,529 B | 99.53 ms | 68.44 ms |
| notifications | 10,000 | 2,447,195 B | 87.09 ms | 68.80 ms |
| automation_events | 10,000 | 2,586,029 B | 103.03 ms | 96.58 ms |

四组数据均低于 32 MiB，单次解析和序列化远低于 5 秒宽松回归上限。生产保留策略会把 Job、通知和事件控制在压力样本以下。

## SQLite 启动门槛

满足任意一项才输出 `prepare_sqlite_migration`：

- 订阅数达到 500；
- Job、通知和自动化事件合计达到 10,000；
- 任一 Store 达到 32 MiB；
- 出现现有内存索引无法满足的复杂查询。

否则输出 `keep_json`。当前实际数据低于门槛，不引入 SQLite 依赖，也不进行长期 JSON/SQLite 双写。

## 未来迁移合同

达到门槛后，迁移必须遵守以下状态机：

1. 冻结迁移输入并保留原始 JSON；
2. 在临时 SQLite 文件中按稳定主键执行可重复导入；
3. 校验每类记录数、主键集合和规范化内容校验和；
4. 校验成功后才进行单次原子切换；
5. 切换前创建可恢复快照，切换失败或启动校验失败时回滚 JSON；
6. 完成后使用 SQLite 单写，不长期双写；JSON 只作为迁移源和备份保留。

这些约束由 `SqliteMigrationContract` 暴露并由测试锁定；在门槛未达到时不创建数据库文件。
